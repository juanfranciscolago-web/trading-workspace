## 3. Bugs detectados y resueltos

### Bug #1: save_proposal omite time_horizon_days

**Cuándo:** descubierto durante B.3.7 al diseñar el E2E test.
**Síntoma:** `trades.proposals.time_horizon_days` siempre NULL en filas saved post-Sprint 3.
**Root cause:** `MessageRepository.save_proposal` insertaba solo 9 columnas; faltaba `time_horizon_days` en el INSERT (el valor sí estaba en el JSONB `full_payload`).
**Fix:** 3 líneas en `src/multi_agent/persistence/message_repository.py` — agregar column al SQL list + placeholder + `msg.thesis.time_horizon_days` al params tuple.
**Backward compat:** filas antiguas (Sprint 3 pre-fix) quedan con NULL. Solo nuevos saves persisten correctamente.
**Verificado:** assertion en E2E test (`list_item["time_horizon_days"] == 35` desde Shape A fixture) + manual sanity script (input=36, persisted=36, match True).
**Commit:** `211daef`.

### Bug #2: ProposalMessage enum case mismatch en fixture de test

**Cuándo:** descubierto durante B.3.6 al escribir tests/api/test_trades_routes.py.
**Síntoma:** pytest collection error → `pydantic_core._pydantic_core.ValidationError: 2 validation errors for ProposalMessage. message_type: Input should be <MessageType.PROPOSAL: 'PROPOSAL'>. agent_id: Input should be 'ATHENA', 'APOLLO', ...`.
**Root cause:** fixture inicial usaba lowercase (`"proposal"`, `"athena"`). El schema Pydantic exige uppercase para `message_type` y `agent_id` enum values.
**Fix:** cambiar `_build_proposal_payload` a `"PROPOSAL"` y `"ATHENA"`. Ajustar el assert correspondiente (`data["agent_id"] == "ATHENA"`).
**Implicación detectada (no fix, tech debt registrado):** inconsistencia interna del schema — `proposing_agent` column en DB se almacena lowercase (`save_proposal` hace `.value.lower()`), pero `agent_id` field en JSONB stays uppercase (Pydantic dump preserves enum value).
**Commit:** `e14eb58`.

### Nueva regla #15 (aprendida hoy)

**#15. Verificar freshness del backend running antes de testar endpoints nuevos. Preferir uvicorn sin `--reload` en sesiones de dev largas.**

**Why:** hoy descubrimos que el backend PID 74908 venía corriendo desde Sat 10AM (pre-B.3.5). Al intentar smoke-test del trigger endpoint, retornó 404 porque el endpoint todavía no existía en ese código. Tomé el 404 por bug del frontend hasta verificar que el problema era backend stale. ~10 min perdidos. Pattern para evitar.

**How to apply:**
- Cuando se agrega/modifica un endpoint backend en un commit, antes de smoke-testar desde frontend o script: `curl -X POST/GET <endpoint>` directo para verificar que el code path está activo (200/422/500 esperado, NO 404).
- Para sesiones de dev largas: restartear el backend al menos al inicio del día con código del último commit. Para sesiones de cambios solo-frontend: NO necesario restart; backend running está OK.
- **Sub-regla:** preferir invocar uvicorn sin `--reload` en sesiones de dev largas. Razón: `--reload` re-corre el lifespan (DB pool, alert workers, telegram bot init) cada vez que detecta cambio. En sesiones con muchos commits backend, esto genera noise + edge cases en lifespan re-runs (telegram bot reconnect timeouts, alert workers race conditions, etc.). `scripts/run_api.sh` tiene `--reload` hardcoded — invocar uvicorn directo sin flag.
